# AI編集を安全に進めるための手順

この文書は、Gemini-cli などの AI ツールで Rails アプリのコードを直すときに、できるだけ壊れにくくするための簡単な手順です。

対象は、開発に慣れていない人です。全部を完璧にやる必要はありません。まずは「大事故を減らす」ことを目的にしてください。

## なぜコードが壊れるのか

AI での編集は便利ですが、次のような理由でファイルが壊れることがあります。

- AI の返答が途中で切れる
- 通信が弱く、編集結果が中途半端に反映される
- 一度に大きく直しすぎる
- 直したあとに確認せず、そのまま保存や commit をしてしまう

特に、通信が弱い環境では次の事故が起きやすくなります。

- 編集途中で接続が切れる
- 再試行したら前の編集の残りが混ざる
- ファイルの末尾に意味不明な文字が残る
- 同じファイルを何度も編集して、差分が壊れる

## まず覚えること

最低限、次の3つだけ覚えてください。

1. 一度に大きく直さない
2. 直したらすぐ確認する
3. main に直接入れる前にテストや CI を見る

この3つだけでも事故はかなり減ります。

## 作業前にやること

### 1. 通信が安定しているか確認する

Wi-Fi が不安定な場所では、大きな編集をしない方が安全です。

避けた方がよい作業:

- 複数ファイルをまとめて大きく変更する
- モデル、コントローラ、ビュー、migration を一度に直す
- 画面全体のリファクタ

通信が不安定なときは、次のような小さい作業に分けてください。

- 文言修正だけ
- 画面1つだけ
- コントローラ1ファイルだけ
- CSS だけ

### 2. 作業前に最新を取り込む

古いコードのまま作業すると、あとで変な壊れ方をしやすくなります。

最低限これをやります。

```bash
git pull
```

### 3. いまの状態を確認する

作業前に、すでに変更が残っていないか確認します。

```bash
git status
```

何か出たら、その変更は今回の作業と関係あるのか確認してください。よく分からないなら、そのまま次へ進まない方が安全です。

## 作業中のルール

### 1. AI への依頼は小さくする

悪い例:

- 注文機能を全部直して
- 集計画面も管理画面も全部まとめて直して

良い例:

- 注文一覧のページ列だけ狭くして
- ユーザー一覧に追加ボタンを出して
- orders_controller のこのエラーだけ直して

依頼が小さいほど、壊れたときの被害も小さくなります。

### 2. 1回ごとに diff を見る

AI が直したあと、そのまま次に進まず差分を見ます。

```bash
git diff
```

見るポイント:

- 関係ないファイルまで変わっていないか
- ファイル末尾に変な文字が入っていないか
- `end` や `)` が不自然に増えていないか
- 大きすぎる変更になっていないか

もし diff が読みにくいほど大きいなら、その変更は一度にやりすぎです。

### 3. 変な挙動が出たら、そのセッションを切る

AI ツールが次のような動きをしたら危険です。

- 同じ説明を何度も繰り返す
- 途中で止まる
- 保存された内容が一部しかない
- 関係ないファイルを直し始める

その場合は無理に続けず、いったん止めてください。通信が不安定なときは特にそうです。

## 作業後にやること

### 1. 最低限の確認をする

Ruby や Rails では、次の確認が有効です。

#### Ruby の文法だけ確認したいとき

```bash
ruby -c app/controllers/orders_controller.rb
```

ファイル名は直したものに合わせて変えてください。

#### Rails のテストを流す

```bash
bin/rails test
```

全部は難しくても、少なくともエラーが出ないかは見てください。

#### RuboCop がある場合

```bash
bin/rubocop
```

### 2. 画面も確認する

CLI の結果が通っても、画面が崩れていることはあります。

確認例:

- 注文一覧が開くか
- 注文入力が開くか
- 追加したボタンが押せるか
- 変更した画面でエラーが出ないか

### 3. CI の結果を見る

push した後は GitHub Actions を見ます。

最低限見るもの:

- `lint`
- `test`
- `deploy`

どれか1つでも落ちたら、その時点では「直った」と考えない方が安全です。

## 通信が弱い環境での注意

通信が弱い環境では、次のやり方をおすすめします。

### やってよい

- 小さい修正
- 1ファイルずつの修正
- 文言変更
- 見た目だけの修正

### 避けた方がよい

- migration を含む大きい変更
- 複数の画面をまとめて変える作業
- deploy 周りの修正
- 自動で大量ファイルを書き換える作業

### 不安定な時のコツ

- 1回の依頼を短くする
- 1回直したらすぐ `git diff` を見る
- 少しでも変ならそのセッションをやめる
- 大きい修正は通信が安定した環境でやる

## 失敗しにくい実際の流れ

迷ったら、この順番で作業してください。

1. `git pull`
2. `git status`
3. AI に小さい修正を1つだけ依頼
4. `git diff`
5. `ruby -c` または `bin/rails test`
6. 画面確認
7. commit
8. push
9. GitHub Actions を確認

## 困ったときの判断

### そのまま続けてよいケース

- 変更ファイルが少ない
- diff が読める
- テストが通る
- 画面も正常

### いったん止めた方がよいケース

- diff が大きすぎる
- 関係ないファイルが変わっている
- ファイルの末尾に変な文字がある
- AI の返答が途中で切れた
- 通信が何度も切れている
- 何を直したのか自分で説明できない

## AI に最初に伝える指示テンプレート

AI は、何も言わないと「直すこと」だけを優先して、「確認してから終わること」を省く場合があります。

そのため、最初に完了条件をはっきり伝える方が安全です。以下のテンプレートをそのまま使えます。

### 基本テンプレート

```text
この修正をしてください。
ただし、次の条件を守ってください。

1. 一度に大きく変えず、必要最小限の変更にする
2. 修正後に git diff を確認する
3. 修正後にテストまたは構文チェックを実行する
4. エラーが残る場合は、完了扱いにしない
5. テストが通っていないならコミットしない
6. 何を変更したかを最後に短く説明する
```

### Rails プロジェクト向けテンプレート

```text
この Rails アプリを修正してください。
作業ルールは次の通りです。

1. 関係ないファイルは変更しない
2. 修正後に git diff を確認する
3. 修正後に bin/rubocop と bin/rails test を実行する
4. 失敗した場合は、その原因を説明する
5. テストや lint が通らないならコミットしない
6. 最後に、変更したファイルと内容を短くまとめる
```

### 通信が弱い環境で使うテンプレート

```text
通信が不安定なので、一度に大きく変更しないでください。

1. 1ファイルまたは1機能ずつ修正する
2. 毎回 git diff で差分を確認する
3. 途中で不自然な変更が出たら作業を止める
4. 修正後は最低限の構文チェックかテストを実行する
5. 問題が残る場合はコミットしない
```

### コミット前確認を強制したいテンプレート

```text
修正後、次の条件を満たした場合のみコミットしてください。

1. git diff を確認済み
2. 変更内容が要求に一致している
3. bin/rubocop が成功している
4. bin/rails test が成功している
5. 失敗がある場合はコミットせず、失敗内容を報告する
```

## AI に依頼するときのコツ

次のような依頼は安全です。

- 注文一覧のページ列だけ狭くしてください
- users の name を一覧に表示してください
- reports_controller のこのエラーだけ直してください

次のような依頼は危険です。

- 全体をいい感じに直してください
- バグを全部直してください
- ついでに設計も整理してください

依頼が広いほど、AI は勝手な判断をしやすくなります。

## 最後に

非エンジニアが AI を使ってコードを直すのは、悪いことではありません。ただし、「AI が直したから大丈夫」と考えるのは危険です。

一番大事なのは次の2つです。

- 小さく直す
- 直したあとに必ず確認する

この2つを守るだけでも、今回のような壊れ方はかなり減らせます。
